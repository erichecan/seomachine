## 一、产品定位

AI 驱动的 SEO 内容创作 SaaS，帮助用户完成关键词研究、文章生成、一键发布到多平台的完整工作流。

**面向用户**：内容运营、独立站站长、SEO 从业者、电商卖家

## 二、发布平台对接方案

### WordPress（直连）

使用 WordPress REST API，用户在 WordPress 后台生成「应用程序密码」粘贴到系统，无需手动上传。

```
POST https://their-site.com/wp-json/wp/v2/posts
Authorization: Basic base64(username:app-password)
Body: { title, content, status, categories, tags }
```

授权方式：应用程序密码（推荐，主流 SaaS 都用这个）

### Shopify（OAuth 直连）

通过 Shopify Partner API + OAuth 完成授权，发布到 Blog Posts。

```
用户点「连接 Shopify」
  → 输入店铺域名（xxx.myshopify.com）
  → 跳转 Shopify 授权页
  → 同意后拿到 Access Token
  → 调用 Admin API 发文
```

需在 Shopify Partner 后台注册 App 获取 Client ID / Secret。

### 用户自建独立站（Webhook）

用户在系统内填写：

```
接口地址：https://their-site.com/api/receive-post
API Key：用户自定义
请求格式：约定 JSON 结构
```

系统按约定格式 POST，用户在自己网站实现接收端。适合有开发能力的用户。

也可提供官方 SDK/插件（npm 包），降低用户接入门槛。

### 未来可扩展

- Webflow（有 CMS API）
- Ghost（有官方 API）
- 其他有 API 的 CMS 平台

## 四、成本控制（Fair Use Policy）

官方说法在服务条款注明：「异常高用量账户可能受到限制」，后台静默执行。行业惯例，Notion、Linear、Figma 均采用此策略。

### 控制手段

**1. 限速（最常用，用户感知最弱）**

每小时 / 每天设定生成上限，超出后提示「今日生成已达上限」。对外话术：「保证内容质量，避免过度依赖 AI」。

**2. 模型降级（最省钱）**

```
每月前 N 篇  → GPT-4o（质量好）
超出后       → GPT-4o mini 或 DeepSeek（成本降 10 倍）
```

用户感知差异极小，成本被有效封顶。

**3. 队列延迟（重度用户自然流失）**

普通用量：实时生成  
超出阈值后：加入队列，提示「预计 2-5 分钟完成」（实为故意延迟）

**4. 内容长度限制**

按套餐限制每篇字数上限，字数直接决定 token 消耗，最精准的成本控制。

**5. 请求缓存**

相同关键词已有人生成过则返回缓存，成本为零。SEO 场景下大词命中率较高。

### 各套餐控制策略组合

```
基础版 $29/月：
  - 每天限 5 篇（软限制）
  - 每篇最多 1500 字
  - 超量后模型自动降级为 DeepSeek

专业版 $79/月：
  - 每天限 20 篇
  - 每篇最多 3000 字
  - 全程 GPT-4o，不降级

团队版 $199/月：
  - 无硬性每日限制
  - 超过 500 篇/月触发人工审核
  - 全程 GPT-4o
```

## 五、成本估算

生成一篇 1500 字文章的 LLM 成本：

| 模型 | 每篇成本 | 备注 |
|------|---------|------|
| DeepSeek V3 | ~$0.01 | 走量首选，中文内容佳 |
| GPT-4o mini | ~$0.03 | 性价比平衡 |
| GPT-4o | ~$0.15 | 高质量输出 |
| Claude Sonnet | ~$0.10 | 高质量输出 |

基础版用户全月 5篇/天 × 30天 = 150篇，用 DeepSeek 成本约 $1.5，毛利极高。
专业版 20篇/天 × 30天 = 600篇，用 GPT-4o 成本约 $90，需配合降级策略控制。

## 六、后台用量追踪（技术要点）

每个用户需记录：

```
- 当月已生成篇数
- 当月已消耗 token 数
- 今日请求次数
- 当前套餐等级
```

每次请求流程：

```
收到生成请求
  → 检查今日请求次数 → 超限则拒绝或加队列
  → 检查本月用量    → 决定调用哪个模型
  → 调用 LLM 生成
  → 更新用量计数（Redis 存储）
  → 返回结果
```

技术实现：Redis 计数器 + 请求中间件拦截，不复杂。

## 七、架构方向

从桌面 App 转为 SaaS 后，核心变化：

```
桌面 App（用户本地持有 API Key）
    ↓
Web App / 轻量客户端
    → 你的后端（统一持有 LLM Key，控制配额，记录用量）
    → LLM API（DeepSeek / GPT-4o）
    → 发布连接器（WordPress / Shopify / Webhook）
```

**LLM Key 必须在服务端，绝不暴露给客户端。**
