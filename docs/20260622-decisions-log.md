## 决策 1：产品形态 — 纯 SaaS Web，放弃桌面 App

**结论**：只做 SaaS Web 版，桌面 App 方向已废弃。

**背景**：
- 原有两个同源仓库：`yuansir/seomachine`（Tauri 桌面 App，中文适配）和 `TheCraigHewitt/seomachine`（Claude Code 工作区）
- 初期考虑两者并行开发

**放弃桌面 App 的原因**：
- 需要用户下载安装，增加使用门槛
- 需要维护 macOS、Windows 多平台安装包
- 版本更新需要用户主动重新下载（无自动更新机制时问题更大）
- 用户兼容性问题（系统版本、架构差异）
- SaaS Web 访问零门槛，功能可随时更新，用户体验更好

**影响**：
- 桌面 App 代码（`src/`、`src-tauri/`）暂时保留但不继续开发
- Phase 1 开始构建 SaaS Web 后端和前端
- 路线图去除原"Phase 2 Desktop App 完善"

## 决策 2：商业模式 — 月度订阅，统一承担 LLM 成本

**结论**：月度订阅制，不让用户提供自己的 API Key，平台统一承担 LLM 成本。

**考虑过的方案**：
- ~~按字数收费~~（用户感知差，不直观）
- ~~让用户填自己的 API Key~~（降低 LLM 成本，但用户体验差，付费转化率低）
- ✅ **月度订阅 + Fair Use**（行业主流，体验好，成本可控）

**理由**：
- 月订降低决策门槛，更容易转化
- 用户无需理解 token 定价，体验更简洁
- 平台控制 LLM Key，可以做多模型路由和成本优化
- Fair Use Policy 是行业惯例（Notion、Linear、Figma 均采用）

## 决策 3：成本控制策略 — Fair Use Policy（静默执行）

**结论**：后台静默执行 Fair Use，服务条款注明异常高用量可能受限，但不对外主动宣传具体规则。

**四种控制手段**（组合使用）：

| 手段 | 触发条件 | 用户感知 |
|------|---------|---------|
| 限速（每日篇数上限） | 超出套餐日限 | 低，提示"今日已达上限" |
| 模型降级 | 超出月度阈值 | 极低，质量差异不明显 |
| 队列延迟 | 重度用户 | 低，提示"预计 2-5 分钟" |
| 内容长度限制 | 按套餐硬性限制 | 无感知，透明规则 |

**各套餐策略**：
```
基础版 $29/月：
  - 每天限 5 篇（软限制）
  - 每篇最多 1500 字
  - 超量后模型自动降级为 DeepSeek

专业版 $79/月：
  - 每天限 20 篇
  - 每篇最多 3000 字
  - 全程 GPT-4o，不降级

团队版 $199/月：
  - 无硬性每日限制
  - 超过 500 篇/月触发人工审核
  - 全程 GPT-4o
```

**成本估算参考**：

| 模型 | 每篇 1500 字成本 |
|------|---------------|
| DeepSeek V3 | ~$0.01 |
| GPT-4o mini | ~$0.03 |
| GPT-4o | ~$0.15 |
| Claude Sonnet | ~$0.10 |

基础版全月极端用量（150 篇 × DeepSeek）：约 $1.5，毛利极高。  
专业版全月极端用量（600 篇 × GPT-4o）：约 $90，需配合降级控制。

## 决策 5：发布平台对接方式

**结论**：支持 WordPress（REST API）、Shopify（Partner OAuth）、自定义 Webhook，暂不支持其他 CMS。

**WordPress**：
- 用户在 WordPress 后台生成「应用程序密码」粘贴到系统
- 调用 `POST /wp-json/wp/v2/posts`，Basic Auth 认证
- 无需插件，WordPress 5.6+ 原生支持

**Shopify**：
- 通过 Shopify Partner App + OAuth 完成授权
- 发布到 Blog Posts，调用 Admin API
- 需要注册 Shopify Partner 账号获取 Client ID/Secret

**自定义 Webhook**（覆盖自研独立站）：
- 用户填写接口地址 + API Key
- 平台 POST 约定格式的 JSON
- 适合有开发能力的用户（如自己用 vibe coding 搭建的站）
- 可提供官方 SDK/npm 包降低接入门槛

**未来可扩展**：Webflow（CMS API）、Ghost（官方 API）

## 决策 6：LLM 策略 — 多模型路由

**结论**：平台统一持有 LLM API Key，绝不暴露给客户端，后端按套餐等级和用量路由到不同模型。

**路由规则**：
```
基础版用户           → DeepSeek V3（成本优先）
专业版 / 团队版用户  → GPT-4o（质量优先）
任意套餐超出月阈值   → 强制降级为 DeepSeek 或 GPT-4o mini
```

**理由**：
- DeepSeek 成本约为 GPT-4o 的 1/15，中文内容质量优秀
- 多模型路由是控制成本的核心杠杆
- 用户感知的质量差异在大多数 SEO 内容场景下极小

## 决策 8：用量追踪方案 — Redis

**结论**：每用户用量数据存储在 Redis，请求中间件拦截。

**追踪字段**：
- 当月已生成篇数
- 当月已消耗 token 数
- 今日请求次数
- 当前套餐等级

**请求流程**：
```
收到生成请求
  → 检查今日请求次数 → 超限则拒绝或加队列
  → 检查本月用量    → 决定调用哪个模型
  → 调用 LLM 生成
  → 更新 Redis 计数器
  → 返回结果
```

## 决策 9：仓库结构 — 合并两个同源仓库

**结论**：以 `yuansir/seomachine`（Tauri 桌面 App）为基础，叠加 `TheCraigHewitt/seomachine`（Claude Code 工作区）的所有工作区资产。

**合并内容**：
- 保留：`yuansir` 的 Rust 代码、Python 脚本、SQLite 数据模块
- 叠加：Craig 版的 `.claude/commands/`（24 个）、`.claude/agents/`（11 个）、`.claude/skills/`（26 个）、`context/`、`data_sources/`

**原因**：
- Craig 版是纯 Claude Code 工作区，无 App 代码；`yuansir` 版有完整 App 代码但工作区资产不如 Craig 版丰富
- 合并后既有可执行的 App 代码基础，又有完整的 AI 工作流能力
